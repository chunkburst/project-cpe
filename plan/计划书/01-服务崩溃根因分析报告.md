# 服务崩溃根因分析报告

## 概述

UDX710 后端服务在运行约3小时后会自行崩溃，即使在空闲状态下也会发生。NCM 模式切换后仅前30秒正常工作随后也会崩溃。本文档深入分析所有根因并提供解决方案。

---

## 根因 #1（致命）：全局互斥锁 `with_serial` 饱和死锁

### 位置
`backend/src/serial.rs:21` - 全局 `DBUS_LOCK` Mutex
`backend/src/dbus.rs:725` - Watchdog 每5秒触发一次

### 问题机制

```
         ┌─────────────────────────────────┐
         │     with_serial() 全局锁         │
         │     Mutex<()> + 5s 超时          │
         └──────────┬──────────────────────┘
                    │
    ┌───────────────┼───────────────┐
    │               │               │
    ▼               ▼               ▼
Watchdog         API请求         SMS/Call
(每5秒)         (用户触发)       Listener
    │               │               │
    ▼               ▼               ▼
check_and_restore  send_at_command  DBus calls
    │
    ▼
find_internet_context ←── 未加锁！直接调用 GetContexts
    │
    ▼
NetworkRegistrationProxy
    │
    ▼
set_data_connection ←── 加锁
```

**死锁过程**：
1. Watchdog 每5秒调用 `check_and_restore_data_connection()`
2. 该函数内部的 `find_internet_context()` **没有经过 `with_serial`**，直接调用 D-Bus
3. 如果此时 ofono 正在处理另一个操作，返回 `InProgress` 错误
4. 该错误导致函数提前返回，但 **已经产生的 D-Bus pending 状态不会清理**
5. 下一次尝试获取 `with_serial` 锁时，上一个操作还没完全释放
6. 5秒锁超时触发 → 操作失败 → 产生更多 pending 状态
7. **正反馈循环**：失败越多，pending 越多，锁越难获取
8. 最终状态累积导致 ofono 服务响应变慢，DBus 调用全部超时

### 影响等级
**致命** - 这是服务崩溃的直接原因。任何 D-Bus 操作失败都会触发正反馈恶化。

---

## 根因 #2（致命）：Watchdog 间隔过短 + iptables 频繁刷新

### 位置
`backend/src/main.rs:216` - `dbus::data_connection_watchdog(conn_clone, 5).await`

### 问题
- Watchdog 每 **5秒** 执行一次完整的数据连接检查
- 每次检查包含：iptables 规则统计 → iptables 清空 → 网络注册状态查询 → context 查询 → 数据连接状态查询
- 每5秒 `iptables -F` 一次，会不断清除 connman gadget tethering 添加的 NAT 规则
- connman 会检测到规则被清空，重新添加 → 与 watchdog 形成 **竞态条件**
- 每次 iptables 操作都是 `spawn_blocking`，创建新线程，资源泄漏

### 影响等级
**致命** - 与根因#1 叠加，形成不可恢复的崩溃条件。

---

## 根因 #3（严重）：`find_internet_context` 未序列化

### 位置
`backend/src/dbus.rs:170-203` - `find_internet_context()`

### 问题
```rust
pub async fn find_internet_context(conn: &Connection) -> zbus::Result<String> {
    // ❌ 没有 with_serial！直接调用 D-Bus
    let proxy = Proxy::new(conn, "org.ofono", "/ril_0", "org.ofono.ConnectionManager").await?;
    let contexts: Vec<...> = proxy.call("GetContexts", &()).await?;
    // ...
}
```

此函数被以下路径调用，**全部未加锁**：
- `check_and_restore_data_connection()` (watchdog)
- `set_data_connection()`
- `get_data_connection_status()`
- `init_data_connection()`

当 `set_data_connection` 正在设置属性（持有锁）时，watchdog 的 `find_internet_context` 尝试调用 `GetContexts`，导致 `org.ofono.Error.InProgress`。

### 影响等级
**严重** - 是根因#1 的触发器之一。

## 根因 #4（严重）：`switch_usb_mode_advanced` 阻塞 tokio 运行时

### 位置
`backend/src/usb_switch.rs:395-543`

### 问题
```rust
pub fn switch_usb_mode_advanced(mode: u8) -> Result<(), String> {
    // ❌ 在 async 上下文中调用 blocking sleep
    std::thread::sleep(std::time::Duration::from_millis(100));    // line 414
    std::thread::sleep(std::time::Duration::from_millis(200));    // line 377
    std::thread::sleep(std::time::Duration::from_millis(1000));   // line 537
    
    // configure_usb_network() 内部还有更多 sleep
    std::thread::sleep(std::time::Duration::from_millis(500));    // line 608
    std::thread::sleep(std::time::Duration::from_millis(100));    // line 616
    std::thread::sleep(std::time::Duration::from_millis(200));    // line 622
    std::thread::sleep(std::time::Duration::from_millis(100));    // line 629
    std::thread::sleep(std::time::Duration::from_millis(300));    // line 635
}
```

总计约 **2.5秒** 的 `thread::sleep`，全部在 tokio 工作线程上执行。

tokio 默认工作线程数 = CPU 核心数（UDX710 通常是2-4核）。如果2个并发的 USB 切换请求同时到达，**所有 tokio 工作线程被阻塞**，导致：
- 所有 API 请求超时
- Watchdog 无法运行
- SMS/Call listener 停止工作
- D-Bus 连接心跳丢失

### 影响等级
**严重** - USB 切换时整个服务暂停。

## 根因 #5（严重）：NCM 模式特有的崩溃问题

### 位置
`backend/src/usb_switch.rs:65-73` + `backend/src/usb_switch.rs:436`

### 问题

NCM 模式配置中 `usb_share_enable: true`，这会导致在 USB 切换过程中发送 AT 指令：
```rust
let _ = set_usb_share_mode(config.usb_share_enable);  // line 436
```

`set_usb_share_mode` 直接往 `/dev/stty_lte30` 写入 AT 指令，**绕过 ofono D-Bus**。与此同时，watchdog 和 ofono D-Bus 可能正在操作 modem。

直接写串口设备与 ofono 的 D-Bus 操作形成 **底层冲突**：
1. ofono 持有 modem 的内部状态锁
2. 直接写串口设备绕过锁，改变 modem 状态
3. ofono 返回的数据与预期不一致
4. ofono 内部状态机混乱
5. 后续 D-Bus 操作全部失败

### 为什么前30秒正常？**
- USB 切换完成后的前30秒，系统网络接口刚建立，connman 还未添加 iptables 规则
- 30秒后 watchdog 第一次触发 → `iptables -F` 清空了 connman 刚添加的规则
- 有规则时 flush → connman 重新添加 → 下一次 flush → 竞态
- 同时 D-Bus 锁竞争开始 → 30秒后所有操作失败

### 影响等级
**严重** - NCM 模式特有，导致切换后30秒崩溃。

## 根因 #6（中等）：MessageStream 资源泄漏

### 位置
`backend/src/sms_listener.rs:221-273`

### 问题
```rust
loop {
    let msg = match stream.next().await {
        Some(Err(e)) => {
            // ❌ 每次错误创建新 MessageStream，旧的不显式 drop
            stream = MessageStream::from(&conn);
            continue;
        }
        None => {
            stream = MessageStream::from(&conn);
            continue;
        }
    };
}
```

`MessageStream` 内部持有 zbus 的内部缓冲区。频繁重建（当 D-Bus 连接不稳定时可能每秒多次）会导致内存增长。

### 影响等级
**中等** - 在长时间运行 + D-Bus 不稳定时累积。

## 根因 #7（中等）：ACTIVE_CALLS HashMap 潜在内存泄漏

### 位置
`backend/src/sms_listener.rs:290-291`

### 问题
```rust
lazy_static! {
    static ref ACTIVE_CALLS: StdMutex<HashMap<String, ActiveCall>> = StdMutex::new(HashMap::new());
}
```

虽然有100条上限，但：
- 如果 ofono 返回的 `CallRemoved` 信号丢失（D-Bus 不可靠传输）
- 对应条目永远不会被清理
- 每条约 200 字节，3小时内可能累积

### 影响等级
**低** - 单独不致命，但与其他问题叠加。

---

## 根因 #8（低）：无任务监督/崩溃恢复

### 位置
`backend/src/main.rs:178-218`

### 问题
所有后台任务（SMS listener、Call listener、init_data_connection、watchdog）使用 `tokio::spawn`，没有任何 panic 捕获或重启机制。如果任何一个任务 panic：
- 整个 tokio 运行时不会崩溃
- 但该任务的功能永久失效
- 用户和日志都不会察觉（除了初始的 panic 日志）

## 修复方案总览

### 紧急修复（阻止崩溃）

| 优先级 | 修复项 | 文件 | 改动 |
|--------|--------|------|------|
| P0 | 增大 watchdog 间隔 | main.rs:216 | 5s → 30s |
| P0 | 移除 watchdog 中的 iptables 刷新 | dbus.rs:726-761 | 删除整个 iptables 检查逻辑 |
| P0 | `find_internet_context` 加锁 | dbus.rs:170 | 包裹 `with_serial` |
| P0 | USB 切换使用 `spawn_blocking` | usb_switch.rs | 整个函数 wrap |
| P1 | 移除 NCM 模式中的 `set_usb_share_mode` | usb_switch.rs | 不再绕过 ofono |
| P1 | `configure_usb_network` 改为异步 | usb_switch.rs | `thread::sleep` → `tokio::time::sleep` |

### 中期加固（提高稳定性）

| 优先级 | 修复项 | 文件 |
|--------|--------|------|
| P2 | 添加任务 panic 恢复 | main.rs |
| P2 | MessageStream 错误重试加退避 | sms_listener.rs |
| P2 | D-Bus 连接健康检查 | dbus.rs |
| P2 | ACTIVE_CALLS 添加定期清理 | sms_listener.rs |

## 参考对比：UDX710-TOOLS

参考项目 https://github.com/LeoChen-CoreMind/UDX710-TOOLS 的实现：
- 使用 shell 脚本直接操作，无异步运行时，无锁竞争
- USB 切换后**不立即刷新 iptables**，依赖系统自动管理
- AT 指令全部通过 ofono D-Bus 发送，不直接写串口
- 无 watchdog 机制，依赖 connman 自动重连

本项目需要在自动化运维（watchdog）和系统稳定性之间找到平衡。
